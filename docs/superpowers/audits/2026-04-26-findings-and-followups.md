## Real bugs surfaced by the audit

### Bug 1 — Silent 422s on every AI-assist insert (Trips 41 + 44)

**What was broken:** The Python orchestrator was sending `origin="ai_assist_manual"` (not in Rails `ORIGINS`) and `category="place"` (not in `CATEGORY_OPTIONS`). Rails rejected every item with a 422 but returned no error body visible to the caller. AI-assist silently wrote nothing.

**Task that surfaced it:** Task 2 (rails_contract source-of-truth module)

**Fix:** `ai_service/app/services/rails_contract.py` (new module); enums corrected in `ai_service/app/services/orchestrator.py`

**Commit:** `5303a76`

---

### Bug 2 — Cross-day item moves were silently no-ops

**What was broken:** `optimize_trip_routing` and `refine_itinerary` were sending `day_plan_id` inside the PATCH update body. Rails `item_params` does not permit that field — it was silently dropped. Items that should have moved to a different day stayed where they were.

**Task that surfaced it:** Task 5 (move_itinerary_item + cross-day wiring)

**Fix:** `ai_service/app/services/rails_client.py` — new `move_itinerary_item` method calling the dedicated `/move` endpoint. `ai_service/app/services/orchestrator.py` — both callers updated to call `move_itinerary_item` for cross-day moves, then send a separate PATCH for `time_slot` only.

**Commit:** `a32e5dc`

---

### Bug 3 — Three AI functions had no manual-mode guard

**What was broken:** `build_trip_itinerary`, `refine_itinerary`, and `add_day_trip` were missing the early-return that `enrich_trip_with_experiences` and `optimize_trip_routing` already had. A stale browser tab or third-party caller could invoke these on a manual-mode trip and the AI would happily overwrite user content.

**Task that surfaced it:** Task 7 (manual-mode isolation tests)

**Fix:** `ai_service/app/services/orchestrator.py` — manual-mode guard added at the top of each of the three functions.

**Commit:** `08c9117`

---

### Bug 4 — Drag-to-add dropped every geo field (Trip 44)

**What was broken:** `handleDragEnd`, `handleAddPlaceToDay`, `ItemForm.handleSubmit`, and `PlaceSuggestions.handleAdd` each hand-built their own payload object. None of them propagated `latitude`, `longitude`, `google_place_id`, `google_rating`, `google_reviews_count`, or `address`. A dragged card arrived in the itinerary with no location data — the map pin never appeared.

**Task that surfaced it:** Task 8 (canonical frontend payload builder)

**Fix:** `frontend/src/utils/itineraryItemPayload.js` (new canonical builder); all four callsites in `frontend/src/pages/TripDetail.js`, `frontend/src/components/itinerary/ItemForm.js`, and `frontend/src/components/trips/PlaceSuggestions.js` replaced with calls to `buildItineraryItemPayload`.

**Commits:** `06fc821` (builder + tests), `a4e3073` (callsite refactor)

---

### Bug 5 — Panel↔map click sync broken when two cards share a name

**What was broken:** `handlePlaceClick` set `highlightedPlaceKey` to `google_place_id || name`. When two unassigned places had the same display name (e.g. two "Coffee Shop" entries), clicking either card highlighted both pins and both panel cards simultaneously.

**Task that surfaced it:** Task 12 (panel/map sync audit)

**Fix:** `frontend/src/components/trips/ExtractedPlacesPanel.js` — passes stable `placeKey` (pool index string) as second arg to `onPlaceClick`. `frontend/src/pages/TripDetail.js` — `handlePlaceClick` prefers the passed `placeKey`, then `place.poolIndex`, then `google_place_id`, then `name` as fallback chain.

**Commit:** `396ba98`

## Surfaces now machine-checked

| Invariant | Test file | What regression it catches |
|---|---|---|
| Rails `ORIGINS`, `CATEGORY_OPTIONS`, and permit-list mirror Python constants exactly | `test_rails_contract.py::TestRailsContractInSync` | Any Rails enum/permit-list change that isn't reflected in `rails_contract.py` |
| Every Python payload builder produces a shape that passes `assert_itinerary_item_payload` | `test_payload_shapes.py` | Any new field added to an orchestrator payload that Rails would reject with 422 |
| All 5 mutating AI functions return early on manual-mode trips | `test_manual_mode_isolation.py` | A future function added without the guard, or a stale caller bypassing it |

| `merge_link_data` does not clobber existing geo-enriched places | `test_race_preservation.py::test_merge_link_preserves_existing_enriched_places` | Link-merge logic regressing to overwrite geocoded data |
| Frontend canonical builder emits every geo field and normalises category/origin | `itineraryItemPayload.test.js` | Any callsite that reverts to a hand-built payload and loses geo fields |

## Follow-ups (not in scope of this audit)

- **`App.test.js` resolver failure** — The pre-existing Jest test for `App.js` fails because `useSmoothScroll` import cannot be resolved. This is unrelated to the audit but worth fixing so `npm test` exits clean.
- **`test_link_merge.py` and `test_reenrich_places.py` are not tagged `contracts`** — These cover real invariants (link-merge idempotency, re-enrichment guard). Consider evaluating whether they should join the contract gate or stay as integration tests.
- **`day_plans` move endpoint not contract-tested** — The audit pinned the `itinerary_item` permit list. The `day_plans` permit list (separate controller) was inventoried in the surface map but has no drift-detection test yet.
- **Links `source_url` dedup** — The surface inventory (`2026-04-26-cross-layer-surface-map.md`) flagged that `itinerary_item_links` dedup logic is undocumented and has no test. A duplicate-link insert could silently succeed or produce a DB unique-constraint violation depending on Rails version.
- **Mobile client contract coverage** — The `mobile/` directory was not in scope. If it builds its own payloads against the same Rails API, those callsites have the same drift risk as the frontend had before this audit.

## How to run the contract suite locally

```bash
# Python contracts (44 tests, ~0.6 s)
cd ai_service
source .venv/bin/activate
pytest -m contracts -v

# All Python tests (164 tests)
pytest tests/ --no-header

# Frontend contract tests (7 tests)
cd frontend
npm test -- --watchAll=false --testPathPattern=itineraryItemPayload
```

CI runs both automatically on every push and PR to `main` via `.github/workflows/contracts.yml`.
